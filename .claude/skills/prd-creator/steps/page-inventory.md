# Dashboard pages (page inventory)

A data app is organized as a set of **dashboard pages**, each supporting a user journey or decision. This phase locks the list of pages — the app's equivalent of "top-level features."

Propose a list of **3–7 pages** that together deliver the core purpose. Derive them from the user journeys: ask (or infer) "what are the main questions or tasks the dashboard supports?" and turn each into a page. Present them as a numbered list in chat, each with a one-line description of its purpose and the audience/decision it serves.

Then ask the user (via AskUserQuestion or free text) which to keep, which to cut, and what's missing. Keep the set focused — a tight, well-sequenced handful of pages beats a sprawling one.

The output of this phase is a **locked page inventory, ordered by development priority** (the order also drives the milestone sequence later). Lock it before moving on.
