# {{ProjectName}} - Fabric Data App

## Project Type

A **Microsoft Fabric Data App**: a "visualization-as-code" web application that runs inside the Fabric portal and queries a Power BI **semantic model** via the Execute DAX Queries REST API. This repo holds two layers:

1. **Semantic model** — authored as source (PBIP/TMDL) under `src/`, validated with the Best Practice Analyzer, and deployed to a Fabric workspace.
2. **Data app** — scaffolded with the [Rayfin CLI](https://www.npmjs.com/package/@microsoft/rayfin) into its own subfolder, built from DAX + Vega-Lite + React, and deployed to Fabric/OneLake with `npx rayfin up`.

See `README.md` for the end-to-end workflow.

There is no report layer — visualization lives entirely in the data app (CSS + Vega-Lite). The semantic model is authored as TMDL and deployed on its own; no `.pbip`/`.Report` is used.

## Semantic Model Conventions

### Naming
- **Tables**: lowercase singular for dimensions (`customer`, `product`), plural for facts (`sales`, `purchases`)
- **Columns**: lowercase with spaces (`customer key`, `order date`, `total amount`)
- **Measures**: Title Case with spaces (`Total Sales`, `YTD Revenue`, `Customer Count`)
- **Keys**: suffix with `key` (`customer key`, `product key`), set `isHidden: true`
- **Parameters**: prefix with `Parameter_` in Title Case (`Parameter_Server`, `Parameter_Database`)
- **TMDL files**: lowercase matching table names (`customer.tmdl`, `sales.tmdl`)

### Model Design
- Star schema: denormalized dimensions, normalized facts
- Import mode for all tables
- Parameterized SQL Server connections via `expressions.tmdl`
- Measures defined in their source fact table TMDL files
- Calendar table as the standard date dimension

### Documentation
- Triple-slash comments (`///`) on all tables, columns, and measures with business context
- Inline comments for complex DAX explaining business logic

### TMDL Formatting
- 2 spaces for TMDL properties, 4 spaces for DAX expressions within triple backticks
- Property order: declaration, documentation (///), properties (type, isKey, isHidden, formatString), source/partition last
- One table per `.tmdl` file in `/definition/tables/`

## Fabric Data App Conventions

### Scaffolding & deployment
- The App item must be created in the Fabric portal **first** (New item → App), then scaffolded locally — `<AppName>` must match that item
- The data app is generated by the Rayfin CLI; never hand-create the web scaffold. Run from the repo root so it lands in an `<AppName>/` subfolder (monorepo alongside `src/`):
  `npm create @microsoft/rayfin@latest -- "<AppName>" --template dataapp --workspace "<WorkspaceName>"`
- Install the agent skill files into the scaffolded app with `npx rayfin ai-files install`, and follow them — they encode the supported authentication handshake, DAX-generation, and visual primitives
- Use **npm** for the data app (`npm install`, `npm run dev`, `npx rayfin …`)
- Iterate locally with `npm run dev` (preview at http://localhost:5173); deploy with `npx rayfin up`
- Connect to a model with its **share link** from the Power BI Service (the URL carries the workspace ID and model ID), e.g. `https://app.powerbi.com/groups/<workspace-id>/modeling/<model-id>/modelView`

### Configuration
- `fabric.yaml` — workspace + semantic-model connection for the app
- `rayfin.yml` — Rayfin backend configuration
- These are owned by the Rayfin scaffold; edit them rather than introducing `.env`-style config

### Visuals are authored as "trios"
Each visual is typically three files that work together:
- `*.dax` — a DAX query with `{{PLACEHOLDER}}` tokens (e.g. `{{YEAR}}`, `{{LEVEL}}`, `{{FILTERS}}`) resolved by string replacement at runtime to support filtering and cross-highlighting
- `*.json` — a Vega-Lite specification for the visual (prefer the template's preconfigured visual primitives: bar/line/area/scatter/pie/donut/heatmap/bubble/waterfall/KPI card)
- `*.tsx` — a React component that binds the DAX query to the Vega-Lite spec

### Theming & format strings (app layer)
- Theming is centralized in the app's `global.css`; make styling changes there so they flow to every card, chart, grid, and tooltip — do not restyle components piecemeal
- Define a column's **format string once** (per the semantic model / column metadata); the template applies it everywhere that column appears (axes, tooltips, data labels, grid cells). Keep currency/percent/number formatting consistent across all visuals

### Validation before deploy
- Validate DAX with the Tabular Editor CLI (or the `tabular-editor` plugin) against the model before wiring it into a visual
- Run the Rayfin Playwright browser validation step to confirm visuals render correctly (not cut off/squished), text is readable, and there are no console errors

## Workflows

- **Adding a table**: create TMDL in `definition/tables/`, add `ref table` in `model.tmdl`, add relationships in `relationships.tmdl`
- **Adding measures**: define in the source fact table TMDL file with `///` documentation
- **Deploying the model**: use the `/fabric-cli` skill for workspace and deployment operations (`fab auth login` first)
- **Adding a data-app visual**: create the `.dax`/`.json`/`.tsx` trio per the Fabric Data App conventions above; validate the DAX, then run `npm run dev` to preview
- **Styling the data app**: edit the app's central `global.css` (or use the `/design-system` skill) — there is no report theme JSON
- **Deploying the data app**: `npx rayfin up`, then open the app in the Fabric portal under your Entra identity
